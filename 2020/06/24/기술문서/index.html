개요 및 목적

메타스플로잇이란 무엇인가?

메타스플로잇 구조와 동작원리

  • 메타스플로잇 아키텍쳐

  • 인터페이스 : 사용자와 상호작용하기 위한 사용자 인터페이스

    • msfconsole
      • 인터넷으로 치면 인터넷 브라우저 콘솔 기반이라 보는데 편하진 않음
      • 모듈이랑 통신정도는 우리가 해야한다
      • 기존의 취약점 코드에 하나를 추가해서 분석을 더 쉽게 한다던가
    • armitage
      • gui 툴
    • msfweb
      • msf gui라는 인터페이스가 있었는데 없어지더니 지금은 없음 pro 버전이 웹기반으로 돌아가는데 프로버전을 의미하는 거임

  • 모듈 : 모듈화된 기능을 정의하는 부분(기능-운영체제-서비스)

  • 라이브러리 : 메타스플로잇의 뼈대 역할을 하는 핵심라이브럴들의 모음

  • 기타 : 프레임워크 플러그인 보조도구 등을 포함

라이브러리

  • Rex: 루비 확장 라이브러리
    • 화살표를 받는 쪽이 코드를 제공하는 것
    • 프레임워크 코어에서는 렉스에서 정의한 코드들을 가져다가 쓴다
    • 자식이 렉스의 아이라 생각하자
    • 루비에서 제공하는 기본적인 라이브러리를 가져다썼다. 의존성 문제를 피하기 위해 추가로 필요한 기능들을 직접 만들었다 이것을 확장 라이브러리라 한다
    • 범용클래스
      • 기본적인 것들 파일 불러오는것 예외처리
    • 스레드 팩토리, 스레드 안전성 및 동기화
      • 루비 기본 스레드는 부족한게 있어서 프레임워크에 필요한 기능들을 추가했다
    • 인코딩
    • 아키텍쳐 서브시스템, 어셈블리
      • 그 아키텍처에 맞게 어셈블리 정의가 다를 수 있는데 그걸 위해
    • 소켓, 프로토콜
      • 통신을 위해
    • 호환성
      • 운영체제 식별을 위함
  • 프레임 워크 코어
    • 모듈과 플러그인에 인터페이스를 제공하기 위한 클래스 모음
    • 렉스 라이브러리에서 정의한 기능을 토대로 구성
    • 렉스는 말 그대로 건물의 뼈대만 제공하는 것이라 생각하자
    • 프레임 워크 코어의 경우 건물의 안을 어떻게 정의 할 것인가
    • 핵심 클래스
      • 범용 클래스
        • 상수나 예외가 기술되어 있다
        • 모듈의 유형 운영체제 시스템 브라우저 종류같은게 정의 되어있다.
      • 프레임워크 문맥
        • 프레임워크 핵심요소, 기능들을 한데 모아놓은 것
      • db 관리자
        • msfconsole 구동할 때 부터 시작해서 모듈 로드 및 실행 취약점 데이터 베이스 등 데이터베이스에 관한 전반적인 것 관리
      • 이벤트 관리
        • 모듈을 쓰다 보면 이벤트가 발생한다
        • 스캐너의 상태를 확인하고 싶다? 이런 문제나 이벤트를 확인할 때 쓴다
  • 프레임워크 베이스
    • 프레임 워크에서 사용하는 세션을 구현
    • 코어의 작업관리를 위한 래퍼 인터페이스 제공
    • 핵심 클래스
      • 세션
        • 하나의 연결 상태,
        • 내가 어떤 시스템을 공격했는데 채널이 열린 상태에서 어떤 명령을 보내고 어떻게 처리할 것인가를 말해주는게 세션이다
          • 이 세션의 종류가 많다
            • 인터프리터라던가 쉘이라던가
          • 렉스는 추상적, 베이스는 핵심 기능 코드들을 적어 놓았다
      • 심플 프레임 워크
        • 개발자 입장에서 봤을 때 개발자가 쉽게 사용할 수 있게 해주는 프레임 워크
        • 심플 프레임 워크 쓰면 코드가 간단하게 구성이 된다

모듈

  • 익스플로잇
    • 시스템 및 응용 프로그램의 취약점을 이용하는 공격 코드 모음
    • local vs remote
      • remote 같은 경우 A-B 공격
        • 웹서버를 직접 공격
        • 네트워크 거리는 멀 수 있음
      • local
        • 취약점이 있는 파일이 공격해버림
    • client side vs server side
      • server side는 공격자가 피해자 서버에 직접 공격하는 거임
        • 망치 들고 가서 부셔버리는 거여
      • client side는 대기 타는 거지
      • 가장 큰 차이는 피해자의 개입, 트리거가 누구냐
    • 플랫폼 - 서비스 - 코드 구조
      • 플랫폼
        • 운영체제
      • 서비스
        • 서비스별로 구별되어 있음
          • 삼바냐 http냐 ftp 취약점인가
      • 코드
        • 해당 서비스의 해당 운영체제를 공격할 코드가 숨겨져 있다
  • auxiliary
    • 페이로드를 필요로 하지 않는 공격 또는 정보 수집을 목적으로 하는 코드 모음
      • 익스플로잇 코드의 경우에는 단계가 있다
      • 포트가 열려있다면 확인 후 코드를 보내 익스플로잇은 공격의 목적이 있음 시스템을 장악하는 것이지
      • 취약점이라 하는 것은 결국에는 자물쇠를 따고 들어가는 거임
        • 따고 들어가서 무엇을 할 것인지도 적혀 있어야함
        • 목적 코드가 페이로드임
    • 페이로드가 필요하지 않는 공격? 공격 목적이 없는 코드?
      • DOS,DDOS 공격
        • 취약점 공격 만으로 시스템을 못쓰게 만들어 버리는 거지
      • 정보보안의 3대 요소 비밀성 무결성 가용성 CIA
        • 익스플로잇은 3개 모두 포함함, 피해자 PC의 비밀 정보를,임의로 건드리면 안되는 것을 건드림, 아예 시스템 날려버림 모두 다르지만
    • 대부분은 가용성을 파괴시키는 코드가 모여있음
    • 정보 수집을 목적으로 하는 것을 더 ㅁ낳이 사용
    • 유형 - 서비스 - 코드 구성
      • pdf docx 파일 만들어 공격
      • scanner gather 요거를 가장 많이 씀
        • 그 하위 폴더를 들어가면 유형에 따른 서비스가 있고 그 하위에 코드가 있다
    • 익스플로잇의 공격 성공률을 높여주기 위한 것
  • post
    • 익스플로잇 성공 후 대상 시스템에 대한 추가 공격을 위한 코드 모음
      • 로컬 익스플로잇에 자주 사용
    • 권한 상승, 정보수집, 파일 업 다운로드 등등
    • 주로 로컬 익스플로잇에 사용
      • pdf안에 공격 코드를 넣음 어떤 페이로드를 넣을까?
      • 헷갈릴 수 있음
    • 플랫폼-유형-코드 구성
